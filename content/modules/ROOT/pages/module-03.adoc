[#qos-classes]
== Understanding Quality of Service (QoS)

Many developers feel that requiring CPU/Memory requests is "hacky" or restrictive. This module proves why *Zero-Requests are dangerous*, especially on bare-metal or constrained environments.

=== The Three QoS Classes

Kubernetes assigns every pod to one of three QoS classes based on its resource configuration:

[cols="2,3,3,2",options="header"]
|===
|QoS Class |Definition |Eviction Priority |Use Case

|*Guaranteed*
|`requests == limits` for all containers
|*Lowest* (protected)
|Critical production workloads

|*Burstable*
|Has requests, but `requests < limits`
|*Medium*
|Most production apps

|*BestEffort*
|*NO requests or limits*
|*Highest* (first to die)
|Dev/test only
|===

[WARNING]
====
*The Danger of BestEffort*:

When a node runs out of memory, Kubernetes evicts pods in this order:

1. BestEffort pods (no requests) - *killed first*
2. Burstable pods exceeding requests
3. Guaranteed pods (only if node is critically low)

*If you deploy without requests, your pod is first in line for termination.*
====

=== How to Check Your Pod's QoS Class

[source,bash,role=execute,subs="+attributes"]
----
oc get pod -n {namespace} -l app=besteffort-app -o jsonpath='{.items[0].status.qosClass}'
----

.Sample Output
[source,textinfo]
----
BestEffort
----

Now check a properly configured pod:

[source,bash,role=execute,subs="+attributes"]
----
oc get pod -n {namespace} -l app=guaranteed-app -o jsonpath='{.items[0].status.qosClass}'
----

.Sample Output
[source,textinfo]
----
Guaranteed
----

*No requests, no limits* = *BestEffort* QoS class = *First to die under pressure*.

=== Part 2: The Squeeze - Launch a Noisy Neighbor

[IMPORTANT]
====
*What Just Happened?*

The `load-generator` dropped from 102m CPU to 45m CPU (*56% performance degradation*) because:

1. `noisy-neighbor` has no CPU limit
2. It's consuming all available CPU on the node
3. `load-generator` is being *CPU throttled* by the Linux kernel

*Real-World Impact*: If this were a production API, response times would spike from 50ms to 500ms+, causing user-facing outages.

[WARNING]
====
*Throttling Explained*:

When a container tries to use more CPU than its *limit*, the Linux CFS (Completely Fair Scheduler) throttles it by pausing the process periodically. This is a per-container mechanism — it has nothing to do with noisy neighbors.

* `0.0` = No throttling (healthy)
* `0.5` = 50% of CPU time is paused (severe degradation)
* `1.0` = 100% throttled (essentially frozen)

The stress pod above is trying to use ~1000m CPU but is capped at 200m — the kernel throttles it to 80%+ of its allowed time.

*This is invisible to application developers* unless they're monitoring kernel metrics!
====

=== Part 4: The Fix - Right-Size Using Historical Data

The right-sizer script you ran in Part 3 has already queried the 95th-percentile CPU and memory usage over the past 7 days and printed the recommended values. The `RIGHT-SIZING RECOMMENDATIONS` table at the end of the script output shows the P95 raw values, the suggested requests and limits, and the exact `oc set resources` command to apply.

[TIP]
====
*Right-Sizing Strategy*:

* *CPU Request*: Set to 95th percentile usage (150m in this example)
* *CPU Limit*: Set to 2-3x the request (300-450m) to allow bursts
* *Memory Request*: Set to 95th percentile + 20% buffer (115Mi)
* *Memory Limit*: Set to 95th percentile + 50% buffer (144Mi)

*Why 95th percentile?*

* 99th percentile includes rare spikes (over-provisioning)
* 50th percentile (median) is too aggressive (frequent throttling)
* 95th percentile balances efficiency with reliability
====

Now let's update the deployment:

[source,bash,role=execute,subs="+attributes"]
----
oc set resources deployment load-generator -n {namespace} \
  --requests=cpu=150m,memory=115Mi \
  --limits=cpu=400m,memory=192Mi
----

Verify the QoS class changed:

[source,bash,role=execute,subs="+attributes"]
----
sleep 10
oc get pod -n {namespace} -l app=load-generator -o jsonpath='{.items[0].status.qosClass}'
----

.Sample Output
[source,textinfo]
----
Burstable
----

[NOTE]
====
To achieve *Guaranteed* QoS, you need `requests == limits`:

[source,bash,subs="+attributes"]
----
oc set resources deployment critical-app -n {namespace} \
  --requests=cpu=200m,memory=256Mi \
  --limits=cpu=200m,memory=256Mi
----

This ensures:
* Pod is protected from eviction
* Pod gets exactly what it requested (no throttling)
* Scheduler knows precise capacity requirements
====

=== Part 5: The Scale - Implement HPA

Now that resources are right-sized, we can implement Horizontal Pod Autoscaling (HPA) safely.

[IMPORTANT]
====
*Why HPA Requires Accurate Requests*:

HPA scales based on *percentage of CPU request*. If your request is:

* Too low: HPA triggers constantly (flapping)
* Too high: HPA never triggers (waste)
* Just right: HPA responds to real load changes

*Formula*: `Target Utilization = (Current CPU / CPU Request) × 100%`
====

Create an HPA targeting 75% CPU utilization:

[source,bash,role=execute,subs="+attributes"]
----
oc autoscale deployment load-generator -n {namespace} \
  --min=1 --max=5 --cpu 75%
----

Verify HPA was created:

[source,bash,role=execute,subs="+attributes"]
----
oc get hpa -n {namespace}
----

.Sample Output
[source,textinfo]
----
NAME             REFERENCE                   TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
load-generator   Deployment/load-generator   64%/75%   1         5         1          10s
----

At 64% utilization (16m / 25m request), the HPA is watching but not yet scaling. Now simulate a traffic spike by scaling the load to 500 connections:

[source,bash,role=execute,subs="+attributes"]
----
oc set env deployment/load-generator -n {namespace} TARGET_RPS=500
----

Wait 90 seconds for the new pod to roll out and wrk to ramp up to 500 connections, then check HPA status:

[source,bash,role=execute,subs="+attributes"]
----
oc get hpa -n {namespace} load-generator
----

.Sample Output
[source,textinfo]
----
NAME             REFERENCE                   TARGETS    MINPODS   MAXPODS   REPLICAS   AGE
load-generator   Deployment/load-generator   140%/75%   1         5         2          2m
----

[TIP]
====
*HPA Scaling Decision*:

```
Current CPU: 140% of request (35m / 25m)
Target: 75%
Desired Replicas: ceil(1 × 140 / 75) = ceil(1.87) = 2
```

HPA scaled from 1 → 2 replicas to bring utilization down to target.

After the second replica stabilizes:
```
Total CPU: ~35m distributed across 2 pods = ~17m per pod
Utilization: 17m / 25m = 68% ✓ Within target
```
====

=== Part 6: Debug OOMKilled Events

Now let's intentionally trigger an Out-Of-Memory kill to see how to debug it.

Check if there are any recent OOMKilled events:

[source,bash,role=execute,subs="+attributes"]
----
oc get events -n {namespace} --field-selector reason=OOMKilling --sort-by='.lastTimestamp' | tail -5
----

If there are no recent OOM events, let's check pod status for past kills:

[source,bash,role=execute,subs="+attributes"]
----
oc get pods -n {namespace} -o json | \
  jq -r '.items[] | select(.status.containerStatuses[]?.lastState.terminated.reason == "OOMKilled") | .metadata.name'
----

To investigate an OOMKilled pod, check its last termination reason:

[source,bash,role=execute,subs="+attributes"]
----
oc get pod <pod-name> -n {namespace} -o jsonpath='{.status.containerStatuses[0].lastState.terminated}'
----

.Sample Output
[source,json]
----
{
  "exitCode": 137,
  "reason": "OOMKilled",
  "message": "Container exceeded its memory limit"
}
----

[IMPORTANT]
====
*OOMKilled Troubleshooting Checklist*:

1. *Check memory limit*: `oc get pod <pod> -o jsonpath='{.spec.containers[0].resources.limits.memory}'`
2. *Check actual usage before kill*: Run the right-sizer script with the pod name to see historical memory usage:
+
[source,bash,subs="+attributes",role=execute]
----
NAMESPACE={namespace} POD_SELECTOR="<pod-name>" ~/module-03/resource-right-sizer.sh
----
3. *Increase memory limit* if usage legitimately exceeds limit
4. *Investigate memory leak* if usage grows unbounded over time

*Exit Code 137* always means OOMKilled (128 + 9 SIGKILL signal).
====

=== The Developer Social Contract

[IMPORTANT]
====
*Resource Requests Are Not Restrictions*:

Many developers resist setting requests because they feel like "quotas" or "limits." This module proves the opposite:

* *Requests = Promise to Scheduler*: "I need this much to run"
* *Limits = Safety Net*: "Don't let me consume more than this"
* *No Requests = Russian Roulette*: "Kill me first when the node is full"

*Accurate requests are a contract between you and the infrastructure team.* Without them:

* The scheduler is blind (poor placement decisions)
* Your pods are first to evict
* HPA doesn't work correctly
* Forecasting is impossible
====

== Key Takeaways

* *BestEffort QoS (no requests) = First to die under node pressure*
* *CPU throttling is invisible to apps but destroys performance*
* *Use Prometheus 95th percentile for right-sizing, not guesswork*
* *HPA requires accurate requests to scale correctly*
* *OOMKilled = Exit Code 137, check memory limits vs actual usage*
