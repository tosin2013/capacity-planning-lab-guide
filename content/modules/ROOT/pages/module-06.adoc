[#scenario]
== The Scenario: Black Friday + AWS Outage

*Date*: Black Friday (the biggest shopping day of the year)
*Time*: 11:45 AM EST
*Your Role*: Platform Engineering On-Call Lead

*Situation*:

Your company runs a multi-tenant e-commerce platform on OpenShift across 3 AWS regions (us-east-1, us-west-2, eu-west-1). Normal daily traffic is 10,000 requests/second. Black Friday traffic typically spikes to 100,000 requests/second.

*11:50 AM*: Traffic starts ramping up. HPAs are scaling pods as expected.

*11:58 AM*: AWS reports "degraded performance" in us-east-1 (your primary region). Two availability zones are experiencing network issues.

*12:00 PM*: 30% of your worker nodes in us-east-1 become unreachable. Pods are evicting. Traffic is failing over to us-west-2, which is now overloaded.

*Your Mission*:

Keep the platform online for the next 60 minutes using the capacity planning skills from Modules 1-5. Every decision has trade-offs:

* Scale up → costs spike, but customers stay happy
* Throttle traffic → save money, but lose sales
* Fail over to another region → avoids outage, but that region might also overload
* Emergency node addition → takes 15 minutes, might be too late

[IMPORTANT]
====
*Game Rules*:

1. *This is a simulation* - we're using pre-deployed workloads and scripts to mimic the scenario
2. *You have limited "budget"* - every node you add costs $200/hour (simulated)
3. *Success criteria*:
   - Keep >95% request success rate for 60 minutes
   - Spend <$5,000 in emergency capacity
   - No customer-facing service goes down completely
4. *You'll work in teams* - platform team vs application team perspectives
5. *Facilitator will inject chaos* - node failures, traffic spikes, etcd pressure
====

=== Wave 2: The Node Failures (T+15 to T+30 minutes)

[IMPORTANT]
====
*Facilitator Action*:

Simulate the AWS availability zone failure:

```bash
# Cordon 30% of worker nodes (simulate unreachable nodes)
# Uses awk ceiling arithmetic — 'bc' truncates 0.9 to 0 on 3-node clusters
TOTAL_NODES=$(oc get nodes -l node-role.kubernetes.io/worker --no-headers | wc -l)
CORDON_COUNT=$(awk -v n="$TOTAL_NODES" 'BEGIN { c = n * 0.3; print (c == int(c)) ? int(c) : int(c)+1 }')

oc get nodes -l node-role.kubernetes.io/worker --no-headers | awk '{print $1}' | head -n "$CORDON_COUNT" | xargs -I {} oc adm cordon {}

# Wait 30 seconds, then drain them (simulate node shutdown)
sleep 30
oc get nodes -l node-role.kubernetes.io/worker --no-headers | awk '{print $1}' | head -n "$CORDON_COUNT" | xargs -I {} oc adm drain {} --ignore-daemonsets --delete-emptydir-data --force --grace-period=30
```

*Expected Effect*:
* 30% of nodes become SchedulingDisabled
* Pods on those nodes evict and reschedule
* Remaining nodes experience CPU/memory pressure
* Some pods may CrashLoopBackOff if they can't get resources
====

==== Your Actions: Triage the Outage

*In a real multi-AZ outage*, you might see:
* 30-50% of nodes unreachable (not responding to API)
* Pods stuck in "Terminating" state (kubelet can't confirm shutdown)

Check pod distribution:

[source,bash,role=execute,subs="+attributes"]
----

----

.Sample Output
[source,textinfo]
----

     18 ip-10-0-155-132.us-east-2.compute.internal
     15 ip-10-0-174-87.us-east-2.compute.internal
     12 ip-10-0-180-92.us-east-2.compute.internal
     10 ip-10-0-191-45.us-east-2.compute.internal
----

[NOTE]
====
*Pod Rebalancing*:

After a node drain, pods don't automatically rebalance. You'll see:
* Some nodes with 18 pods
* Some nodes with 10 pods

*This is normal.* Kubernetes scheduler only runs when pods are created/rescheduled, not continuously.

*Should you manually rebalance?* Usually no - it causes unnecessary churn. Only rebalance if:
* Imbalance is extreme (>2× difference)
* You're hitting maxPods limits on some nodes
* Performance is visibly degraded
====

Check for pod evictions:

[IMPORTANT]
====
*Decision Point #2*:

Current state:
* 1 node cordoned (20% capacity loss if you have 5 nodes)
* All critical-app pods rescheduled successfully
* Cluster is stable but running at higher utilization

*Options*:

*Option A*: Uncordon the node (assume AWS fixed the issue)
*Option B*: Leave it cordoned and add more nodes (safest, but costs $200/hour)
*Option C*: Wait and monitor (risk: if another node fails, you're in trouble)

*Team Discussion* (3 minutes): What's your call?

*Facilitator Guidance*:
* In a real incident, you'd coordinate with AWS support
* Typical recovery time for AZ issues: 30-120 minutes
* Platform teams lean toward "leave it cordoned until AWS confirms recovery"
* App teams want "uncordon ASAP to save money"

*Option D*: Scale down non-critical workloads to reduce object churn

* Option C is a last resort (only if etcd >7GB and compaction isn't running automatically)
* Option D is good hygiene but won't solve the immediate problem

[NOTE]

*HPA Scale-Down Delay*:

HPA waits 5 minutes by default before scaling down (configurable via `--horizontal-pod-autoscaler-downscale-stabilization-window`).

*Why the delay?*
* Prevents flapping (scale up/down/up/down)
* Gives time to confirm traffic decrease is sustained
* Reduces pod churn (which creates etcd events)

*This is healthy behavior.* Don't manually scale down unless you're certain the spike is over.
