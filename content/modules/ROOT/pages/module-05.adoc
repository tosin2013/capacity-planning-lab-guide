[#rhacm-observability]
== Understanding RHACM Multi-Cluster Observability

Red Hat Advanced Cluster Management provides a centralized observability platform for monitoring dozens or hundreds of OpenShift clusters from a single pane of glass.

=== The Architecture

RHACM Observability consists of:

[cols="2,3,3",options="header"]
|===
|Component |Purpose |Location

|*Thanos*
|Long-term metric storage and querying
|Hub cluster

|*Grafana*
|Dashboard visualization
|Hub cluster

|*Prometheus*
|Metric collection per cluster
|Each managed cluster

|*metrics-collector*
|Forwards metrics from managed → hub
|Each managed cluster

|*observability-alertmanager*
|Centralized alerting
|Hub cluster
|===

[IMPORTANT]
====
*The Data Flow*:

```
Managed Cluster 1 (Prometheus) ──┐
                                  │
Managed Cluster 2 (Prometheus) ──┼──→ Thanos (Hub) ──→ Grafana (Hub)
                                  │
Managed Cluster 3 (Prometheus) ──┘
```

*Key Benefits*:

1. *Unified Query Language*: Use PromQL to query metrics across all clusters simultaneously
2. *Cost Efficiency*: Managed clusters only forward a subset of metrics (configurable via allowlists)
3. *Historical Analysis*: Thanos stores metrics for 30+ days (vs Prometheus' 15 days)
4. *Cross-Cluster Correlation*: Compare capacity trends across dev/staging/prod environments
====

=== The Default Metrics Problem

By default, RHACM Observability collects *all* Prometheus metrics from managed clusters. For a cluster with 1000 pods, this can be:

[source,text]
----
Estimated metric volume per cluster:
- 500,000 time series (pods × metrics per pod)
- 2 samples/min per time series
- 30-day retention

Total: 500K × 2 × 60 × 24 × 30 = 43 billion samples
Storage: ~500GB per cluster per month (uncompressed)
----

[WARNING]
====
*The Observability Cost Explosion*:

If you manage 20 clusters and collect all metrics:

```
20 clusters × 500GB/month = 10TB/month observability storage
```

*Most of these metrics are useless for capacity planning* (detailed pod network stats, individual container CPU usage by core, etc.).

*Solution*: Use metric allowlists to forward only the 50-100 metrics you actually need for capacity dashboards.
====

=== The Capacity Metric Allowlist

For capacity planning, you only need:

[cols="2,3",options="header"]
|===
|Metric Family |Example Metrics

|*Node Capacity*
|`kube_node_status_allocatable`, `node_memory_MemTotal_bytes`

|*Pod Requests*
|`kube_pod_container_resource_requests`

|*Actual Usage*
|`container_cpu_usage_seconds_total`, `container_memory_working_set_bytes`

|*etcd Health*
|`etcd_mvcc_db_total_size_in_bytes`, `etcd_server_has_leader`

|*Deployment Scale*
|`kube_deployment_spec_replicas`, `kube_deployment_status_replicas_available`

|*HPA Metrics*
|`kube_horizontalpodautoscaler_status_current_replicas`
|===

[TIP]
====
*Estimated Storage Reduction*:

* Default (all metrics): 500GB/cluster/month
* Capacity-focused allowlist: 25GB/cluster/month

*Savings*: 95% reduction in observability storage costs!
====

== Why Multi-Cluster Observability Matters for Capacity

Consider this real-world scenario:

[source,text]
----
Your organization has:
- 5 production clusters (US-East, US-West, EU, APAC, Edge)
- 3 staging clusters
- 10 dev clusters

Total: 18 clusters

Question: "How many worker nodes do we need to add next quarter across all environments?"
----

*Without RHACM Observability*:
* Log into 18 separate Prometheus instances
* Run the Pod Velocity query 18 times
* Export to Excel, manually aggregate
* Miss clusters you forgot about

*With RHACM Observability*:
* Single PromQL query: `sum by (cluster) (rate(kube_pod_created[30d]))`
* One Grafana dashboard shows all 18 clusters
* Automated alerting when any cluster approaches capacity

[IMPORTANT]
====
*The Fleet Mindset Shift*:

Capacity planning at scale requires thinking about clusters as *cattle, not pets*:

* You can't manually SSH into each cluster
* You need automated discovery of new clusters
* Standardized dashboards must work for all clusters
* Capacity policies should be fleet-wide, with cluster-specific overrides

RHACM Observability is the foundation for this fleet-centric approach.
====

=== Part 3: Customize the Metric Allowlist

Now let's reduce observability storage costs by only collecting metrics we need for capacity planning.

Check the current metric allowlist:

[source,bash,role=execute]
----
oc get configmap observability-metrics-custom-allowlist -n open-cluster-management-observability -o yaml
----

.Sample Output
[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: observability-metrics-custom-allowlist
  namespace: open-cluster-management-observability
data:
  metrics_list.yaml: |
    names:
      - :node_memory_MemAvailable_bytes:sum
      - cluster:usage:resources:sum
      - cluster_infrastructure_provider
      - cluster_version
    matches:
      - __name__="up"
----

[IMPORTANT]
====
*How Allowlists Work*:

RHACM supports two types of metric filters:

1. *names*: Exact metric name matches (e.g., `cluster_version`)
2. *matches*: Regex patterns (e.g., `__name__="container_.*"` matches all container metrics)

*If the ConfigMap exists*, only the metrics in the allowlist are forwarded. *If it doesn't exist*, all metrics are forwarded (expensive!).
====

Let's add capacity-specific metrics to the allowlist:

[source,bash,role=execute]
----
cat <<EOF | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: observability-metrics-custom-allowlist
  namespace: open-cluster-management-observability
data:
  metrics_list.yaml: |
    names:
      # Cluster metadata
      - cluster_version
      - cluster_infrastructure_provider

      # Node capacity metrics
      - kube_node_status_allocatable
      - kube_node_status_capacity
      - node_memory_MemTotal_bytes

      # Pod resource requests (for capacity planning)
      - kube_pod_container_resource_requests
      - kube_pod_container_resource_limits

      # Deployment scale
      - kube_deployment_spec_replicas
      - kube_deployment_status_replicas_available
      - kube_deployment_created

      # HPA metrics
      - kube_horizontalpodautoscaler_spec_target_metric
      - kube_horizontalpodautoscaler_status_current_replicas
      - kube_horizontalpodautoscaler_status_desired_replicas

      # etcd health
      - etcd_mvcc_db_total_size_in_bytes
      - etcd_server_has_leader

      # Actual resource usage (for right-sizing analysis)
      - container_cpu_usage_seconds_total
      - container_memory_working_set_bytes

      # Node resource usage
      - node_cpu_seconds_total
      - node_memory_MemAvailable_bytes

    matches:
      # Include all "up" metrics (cluster health)
      - __name__="up"

      # Include all kube-state-metrics pod metrics
      - __name__=~"kube_pod_.*"

      # Include CPU throttling metrics (Module 3)
      - __name__="container_cpu_cfs_throttled_seconds_total"
EOF
----

.Sample Output
[source,textinfo]
----
configmap/observability-metrics-custom-allowlist configured
----

[WARNING]
====
*Allowlist Changes Take Effect Gradually*:

After updating the ConfigMap:

1. The `metrics-collector-deployment` on each managed cluster detects the change (~2 minutes)
2. It restarts with the new allowlist (~1 minute)
3. New metrics start flowing to Thanos (~30 seconds)
4. Old metrics (not in the allowlist) stop being collected immediately

*Total delay*: ~3-4 minutes before dashboards reflect the change.

*To force immediate reload*:
```bash
oc rollout restart deployment metrics-collector-deployment -n open-cluster-management-addon-observability
```
(Run this on each managed cluster, or wait for automatic detection.)

=== Part 4: Build a Multi-Cluster Capacity Dashboard

Now let's create a custom Grafana dashboard that shows capacity metrics across all clusters.

In the RHACM Grafana UI:

1. Click "+" → "Create Dashboard"
2. Click "Add new panel"

==== Panel 1: Pod Count by Cluster

Set the following:

* *Panel Title*: "Total Pods by Cluster"
* *Visualization*: Bar gauge
* *Query*:

[source,promql]
----
sum by (cluster) (kube_pod_info)
----

* *Legend*: `{{cluster}}`
* *Unit*: "short" (pod count)
* *Thresholds*:
  - Green: 0-5000
  - Yellow: 5000-8000
  - Red: 8000+ (approaching etcd limit from Module 4)

Click "Apply" to save the panel.

==== Panel 2: CPU Request vs Usage (Fleet-Wide)

Add a new panel with:

* *Panel Title*: "CPU: Requested vs Used (All Clusters)"
* *Visualization*: Time series
* *Query A* (Requested):

[source,promql]
----
sum(kube_pod_container_resource_requests{resource="cpu"})
----

* *Query B* (Used):

[source,promql]
----
sum(rate(container_cpu_usage_seconds_total{container!=""}[5m]))
----

* *Legend*:
  - Query A: "Requested CPU"
  - Query B: "Actual CPU Usage"
* *Unit*: "cores"

Click "Apply".

==== Panel 3: Memory Request vs Usage (Fleet-Wide)

Add a new panel with:

* *Panel Title*: "Memory: Requested vs Used (All Clusters)"
* *Visualization*: Time series
* *Query A* (Requested):

[source,promql]
----
sum(kube_pod_container_resource_requests{resource="memory"})
----

* *Query B* (Used):

[source,promql]
----
sum(container_memory_working_set_bytes{container!=""})
----

* *Legend*:
  - Query A: "Requested Memory"
  - Query B: "Actual Memory Usage"
* *Unit*: "bytes" (Grafana will auto-format to GB)

Click "Apply".

==== Panel 4: Node Runway (Days Until Full)

Add a new panel with:

* *Panel Title*: "Capacity Runway by Cluster (Days)"
* *Visualization*: Table
* *Query*:

[source,promql]
----
# Days until CPU capacity is exhausted based on 30-day growth rate
(
  sum by (cluster) (kube_node_status_allocatable{resource="cpu"})
  -
  sum by (cluster) (kube_pod_container_resource_requests{resource="cpu"})
)
/
(
  sum by (cluster) (increase(kube_pod_container_resource_requests{resource="cpu"}[30d]) / 30)
)
----

* *Legend*: `{{cluster}}`
* *Unit*: "days"
* *Thresholds*:
  - Red: <30 days
  - Yellow: 30-90 days
  - Green: >90 days

Click "Apply".

[TIP]
====
*Understanding the Runway Query*:

This query calculates:

```
(Available Capacity - Current Usage) / Growth Rate = Days Until Full
```

Example:
* Available CPU: 100 cores
* Current requests: 60 cores
* 30-day growth: +5 cores

```
(100 - 60) / (5/30) = 40 / 0.167 = 240 days runway
```

*Caveats*:
* Assumes linear growth (Module 2 explains why Pod Velocity is better)
* Doesn't account for seasonal spikes
* Requires at least 30 days of historical data

*Use this as a quick health check, not a replacement for detailed forecasting.*
====

==== Panel 5: Pod Density by Cluster

Add a new panel with:

* *Panel Title*: "Average Pod Density (pods/node)"
* *Visualization*: Stat
* *Query*:

[source,promql]
----
sum by (cluster) (kube_pod_info) / sum by (cluster) (kube_node_info{node_role_kubernetes_io_worker=""})
----

* *Legend*: `{{cluster}}`
* *Unit*: "short"
* *Thresholds*:
  - Green: 0-150 (healthy utilization)
  - Yellow: 150-200 (approaching maxPods defaults)
  - Red: 200+ (need to tune maxPods or add nodes)

Click "Apply".

==== Panel 6: etcd Database Size by Cluster

Add a new panel with:

* *Panel Title*: "etcd Database Size (MB)"
* *Visualization*: Bar gauge (horizontal)
* *Query*:

[source,promql]
----
max by (cluster) (etcd_mvcc_db_total_size_in_bytes) / 1024 / 1024
----

* *Legend*: `{{cluster}}`
* *Unit*: "MB"
* *Thresholds*:
  - Green: 0-4096 (0-4GB, healthy)
  - Yellow: 4096-6144 (4-6GB, monitor)
  - Red: 6144+ (6GB+, critical - approaching 8GB limit from Module 4)

Click "Apply".

*What This Proves*:

The `cluster` label is automatically added to all metrics by RHACM Observability. This allows you to:

* Query all clusters at once: `sum(kube_pod_info)`

If you only see `{cluster="local-cluster"}`, it means no managed clusters are registered with RHACM. In a production fleet, you'd see dozens of entries here.

=== Part 7: Showback/Chargeback Reporting

One powerful use case for multi-cluster observability is *showback* - reporting resource consumption back to application teams or business units.

Create a new dashboard panel:

* *Panel Title*: "CPU Cost by Namespace (Top 10)"
* *Visualization*: Pie chart
* *Query*:

[source,promql]
----
topk(10,
  sum by (namespace) (
    avg_over_time(kube_pod_container_resource_requests{resource="cpu"}[30d])
  )
)
----

* *Legend*: `{{namespace}}`
* *Unit*: "cores"

Now add a second panel:

* *Panel Title*: "Monthly Cost by Namespace (Estimated)"
* *Visualization*: Table
* *Query*:

[source,promql]
----
sum by (namespace, cluster) (
  avg_over_time(kube_pod_container_resource_requests{resource="cpu"}[30d])
) * 50  # Assume $50/core/month (adjust to your actual rate)
----

* *Legend*: Custom fields:
  - `{{namespace}}` → "Namespace"
  - `{{cluster}}` → "Cluster"
  - Value → "Monthly Cost (USD)"
* *Unit*: "currency (USD)"

[TIP]
====
*How to Calculate Your Real Cost Per Core*:

```
Total Monthly Cloud Bill / Total Allocatable Cores = $/core/month
```

Example:
* Monthly bill: $50,000
* Total cores across all clusters: 1000 cores

```
$50,000 / 1000 = $50/core/month
```

*For on-premise bare-metal*:
* Amortize hardware cost over 3 years
* Add maintenance, power, cooling, data center rent
* Divide by total core-years

Example:
* Server cost: $10,000 (64 cores)
* 3-year lifespan
* $2,000/year maintenance

```
($10,000 / 3 + $2,000) / 64 = $62/core/year = $5.17/core/month
```

*This is a simplified model.* For production FinOps, also factor in:
* Storage costs (PVs)
* Network egress costs
* Load balancer costs
* Overhead allocation (control plane, monitoring, logging)
