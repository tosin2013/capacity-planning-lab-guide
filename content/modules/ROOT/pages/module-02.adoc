== Why Linear Models Fail

Traditional capacity planning uses linear extrapolation:

[source,text]
----
If CPU usage grows 5% per month, in 12 months we'll need 60% more CPU.
----

*This breaks for microservices because:*

* New services deploy in bursts (not linear growth)
* Resource requests vary wildly per service
* Seasonal traffic patterns create spikes
* Deployment velocity (new pods/month) matters more than raw CPU trends

=== The Pod Velocity Model

Instead of tracking CPU percentage, track *how many pods are being deployed over time*:

[source,text]
----
Pod Velocity = Rate of pod creation over time
----

If your organization deploys 50 new microservices per quarter, and each requires an average of 3 replicas with 200m CPU each, you need:

[source,text]
----
50 services × 3 replicas × 200m CPU = 30,000m CPU (30 cores) per quarter
----

This is far more accurate than linear CPU trending because it accounts for *workload growth patterns*.

[IMPORTANT]
====
*The Formula*:

```
Quarterly Node Requirement = (Pod Velocity × Avg Replicas × Avg CPU Request) / Node Allocatable CPU
```

Example:
- Pod Velocity: 50 new services/quarter
- Avg Replicas: 3
- Avg CPU Request: 200m (0.2 cores)
- Node CPU: 8 cores allocatable

```
(50 × 3 × 0.2) / 8 = 30 / 8 = 3.75 nodes per quarter
```

Round up → *You need 4 new worker nodes every quarter.*
====

== Lab 2: The Pod Velocity Calculator & ACM Dashboard

In this lab, you'll calculate pod velocity using PromQL, build forecasting formulas, and create a custom RHACM dashboard to visualize predictions across your fleet.

=== Part 1: Access OpenShift Prometheus

First, let's access the built-in Prometheus monitoring:

[source,bash,role=execute]
----
oc get route -n openshift-monitoring prometheus-k8s -o jsonpath='{.spec.host}'
----

.Sample Output
[source,textinfo]
----
prometheus-k8s-openshift-monitoring.{cluster_domain}
----

[NOTE]
====
Open this URL in a new browser tab. You'll be prompted to log in with your OpenShift credentials. Click "Log in with OpenShift" and use your cluster admin token.
====

=== Part 2: Calculate 30-Day Pod Growth Rate

In the Prometheus UI, enter this PromQL query to calculate pod creation velocity:

[source,promql]
----
rate(kube_pod_created[30d])
----

This shows the rate of pod creation over the last 30 days.

[TIP]
====
*Understanding the Query*:

* `kube_pod_created` - Timestamp when each pod was created
* `rate([30d])` - Calculate the per-second rate over 30 days
* Result is in "pods per second" - we'll convert to "pods per month"
====

Now let's get the total count of new pods in the last 30 days:

[source,promql]
----
increase(kube_pod_created[30d])
----

.Sample Output
[source,textinfo]
----
{namespace="capacity-workshop"} 145
{namespace="openshift-monitoring"} 23
{namespace="openshift-gitops"} 12
...
----

[WARNING]
====
This includes *all* pod creations, including restarts and scaling events. For accurate forecasting, we need to filter for *new Deployments*, not just pod churn.
====

=== Part 3: Track New Deployment Velocity

A better metric is tracking *new Deployments* rather than individual pods:

[source,promql]
----
count(kube_deployment_created{namespace="{namespace}"})
OR
count(kube_deployment_created{namespace="{namespace}"} offset 30d)
----

This compares current deployments to 30 days ago.

Now let's calculate the deployment growth rate:

[source,promql]
----
(
  count(kube_deployment_spec_replicas{namespace="{namespace}"})
  -
  count(kube_deployment_spec_replicas{namespace="{namespace}"} offset 30d)
) / 30
----

.Sample Output
[source,textinfo]
----
Value: 1.2
----

This means you're adding approximately *1.2 new deployments per day* in the `{namespace}` namespace.

=== Part 4: Calculate Average Resource Requests

Now we need to know the average CPU/Memory request per deployment:

[source,promql]
----
avg(
  kube_pod_container_resource_requests{resource="cpu", namespace="{namespace}"}
)
----

.Sample Output
[source,textinfo]
----
Value: 0.15 (150m CPU average per container)
----

Similarly for memory:

[source,promql]
----
avg(
  kube_pod_container_resource_requests{resource="memory", namespace="{namespace}"}
) / 1024 / 1024 / 1024
----

.Sample Output
[source,textinfo]
----
Value: 0.25 (256Mi average per container)
----

=== Part 5: Build the Forecasting Formula

Now combine these metrics into a forecasting dashboard panel. Create a new query that projects node requirements for the next quarter:

[source,promql]
----
# Pod Velocity (new deployments per day)
scalar(
  (count(kube_deployment_spec_replicas{namespace="{namespace}"})
   - count(kube_deployment_spec_replicas{namespace="{namespace}"} offset 30d)) / 30
)
*
# Days in quarter
90
*
# Average replicas per deployment
scalar(avg(kube_deployment_spec_replicas{namespace="{namespace}"}))
*
# Average CPU request per pod
scalar(avg(kube_pod_container_resource_requests{resource="cpu", namespace="{namespace}"}))
/
# Node allocatable CPU (assuming 8 core nodes)
8
----

[IMPORTANT]
====
*Breaking Down the Formula*:

1. *Pod Velocity*: `(current deploys - deploys 30d ago) / 30` = deployments/day
2. *Quarterly Growth*: velocity × 90 days
3. *Total Pods*: quarterly growth × avg replicas
4. *Total CPU Needed*: total pods × avg CPU request
5. *Nodes Required*: total CPU / node CPU capacity

*Result*: Number of nodes you'll need to add this quarter.
====

=== Part 6: Transition to RHACM Grafana

Now that you have the PromQL formula, let's build it into a centralized RHACM dashboard that works across multiple clusters.

Access the RHACM Grafana instance:

[source,bash,role=execute]
----
echo "RHACM Grafana: {grafana_url}"
----

Open this URL in a new tab and log in with your OpenShift credentials.

=== Part 7: Create Pod Velocity Forecast Panel

In Grafana:

1. Click "+ Create" → "Dashboard"
2. Click "Add visualization"
3. Select "Prometheus" as the data source
4. Enter this query:

[source,promql]
----
# Simplified version for multi-cluster
sum(rate(kube_pod_created[30d])) by (cluster) * 2592000
----

This calculates monthly pod creation rate across all clusters.

5. Set the panel title: *"Pod Velocity Forecast - Monthly Growth"*
6. Change visualization type to *"Stat"*
7. Set unit to *"pods/month"*
8. Click "Apply"

=== Part 8: Add Node Requirement Projection

Add a second panel to show projected node requirements:

1. Click "Add panel" → "Add visualization"
2. Enter this query:

[source,promql]
----
# Project quarterly node requirements
ceil(
  (
    sum(rate(kube_pod_created[30d])) by (cluster) * 7776000  # 90 days in seconds
    *
    scalar(avg(kube_pod_container_resource_requests{resource="cpu"}))
  ) / 8  # Assuming 8-core nodes
)
----

3. Set panel title: *"Projected Nodes Needed (Next Quarter)"*
4. Visualization: *"Gauge"*
5. Set thresholds:
   - Green: 0-5 nodes
   - Yellow: 5-10 nodes
   - Red: 10+ nodes

6. Click "Apply"

=== Part 9: Validate the Forecast

Let's test our forecast against actual data. Run this command to count current pods:

[source,bash,role=execute]
----
oc get pods -n {namespace} --no-headers | wc -l
----

Now check what the forecast predicted 30 days ago vs reality:

[source,bash,role=execute]
----
echo "Current pod count: $(oc get pods -n {namespace} --no-headers | wc -l)"
echo ""
echo "Forecast predicted (based on 30-day velocity):"
echo "This would require manual calculation from historical data"
----

[TIP]
====
*Facilitator Discussion Point*:

In production, you'd store daily snapshots of these metrics in a time-series database (Thanos/Prometheus long-term storage) to validate forecast accuracy over time.

*Accuracy improves with*:

* Longer historical data (90 days > 30 days)
* Seasonal adjustment (Black Friday, tax season, etc.)
* Deployment pattern clustering (some teams deploy weekly, others quarterly)
====

=== Real-World Application

Take this model back to your organization:

1. *Capture your deployment cadence* - How many new services per quarter?
2. *Measure average resource requests* - What's your "standard microservice footprint"?
3. *Calculate node runway* - When will you run out of capacity?
4. *Build forecasting into your change approval process* - Every new service deployment should update the forecast

[IMPORTANT]
====
*The Forecasting Pitfall*:

Forecasting is only as good as your *resource requests*. If developers over-request by 10x, your forecast will tell you to buy 10x more hardware than you actually need.

This is why Module 3 (Developer Track - Right-Sizing) is critical. You must fix request accuracy BEFORE forecasting, or you'll forecast waste.
====

* *Pod Velocity (deployments/time) is more predictive than linear CPU trending for microservices*
* *PromQL enables custom forecasting models tailored to your deployment patterns*
* *RHACM provides fleet-wide capacity visibility across multiple clusters*
* *Forecasting accuracy depends on accurate resource requests* (Module 3 fixes this)
