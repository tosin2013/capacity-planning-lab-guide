== Why Linear Models Fail

Traditional capacity planning uses linear extrapolation:

[source,text]
----
If CPU usage grows 5% per month, in 12 months we'll need 60% more CPU.
----

*This breaks for microservices because:*

* New services deploy in bursts (not linear growth)
* Resource requests vary wildly per service
* Seasonal traffic patterns create spikes
* Deployment velocity (new pods/month) matters more than raw CPU trends

=== The Pod Velocity Model

Instead of tracking CPU percentage, track *how many pods are being deployed over time*:

[source,text]
----
Pod Velocity = Rate of pod creation over time
----

If your organization deploys 50 new microservices per quarter, and each requires an average of 3 replicas with 200m CPU each, you need:

[source,text]
----
50 services × 3 replicas × 200m CPU = 30,000m CPU (30 cores) per quarter
----

This is far more accurate than linear CPU trending because it accounts for *workload growth patterns*.

[IMPORTANT]
====
*The Formula*:

```
Quarterly Node Requirement = (Pod Velocity × Avg Replicas × Avg CPU Request) / Node Allocatable CPU
```

Example:
- Pod Velocity: 50 new services/quarter
- Avg Replicas: 3
- Avg CPU Request: 200m (0.2 cores)
- Node CPU: 8 cores allocatable

```
(50 × 3 × 0.2) / 8 = 30 / 8 = 3.75 nodes per quarter
```

Round up → *You need 4 new worker nodes every quarter.*
====

[TIP]
====
*Understanding what the script queries*:

1. *Pod velocity* — pods whose `kube_pod_start_time` falls within the last 30 days, divided by 30 to get pods/day
2. *Deployment count* — `kube_deployment_spec_replicas` (how many Deployments exist)
3. *Avg CPU request* — `kube_pod_container_resource_requests{resource="cpu"}` averaged across all containers
4. *Avg memory request* — same metric for memory, converted to GiB
5. *Node forecast* — `(velocity × 90 days × avg_cpu) / node_allocatable_cpu`, rounded up

Want to see the raw PromQL? Open `pod-velocity-calculator.sh` in your editor.

=== Part 3: Adjust for Your Node Size

Rerun the calculator with the actual allocatable CPU for your worker nodes:

[source,bash,subs="+attributes",role=execute]
----
NODE_ALLOC_RAW=$(oc get node -l node-role.kubernetes.io/worker \
  -o jsonpath='{.items[0].status.allocatable.cpu}')
# Convert millicores (e.g. 7500m) to cores (7.5); leave whole-core values unchanged
NODE_ALLOC=$(python3 -c "
v='${NODE_ALLOC_RAW}'
if v.endswith('m'):
    print(round(int(v[:-1]) / 1000, 2))
else:
    print(round(float(v), 2))
")
echo "Worker node allocatable CPU: ${NODE_ALLOC} cores"
----

Then rerun with the real value:

[source,bash,subs="+attributes",role=execute]
----
cd ~/module-02
NAMESPACE={namespace} NODE_CPU=${NODE_ALLOC} ./pod-velocity-calculator.sh
----

Compare the "Nodes needed" figure to the default (8-core assumption). Does a larger node size reduce your projected node count significantly?

=== Part 5: View the Dashboard

Open the URL printed by the script in a new browser tab. You will see three panels:

1. *Pod Velocity Forecast — Monthly Growth* (Stat panels, one per cluster) — how many pods are being created each month across your fleet
2. *Projected Nodes Needed (Next Quarter)* (Gauge panels, one per cluster) — the computed node requirement with green/yellow/red thresholds
3. *Pod Count Over Time* (Time series) — a historical view of pod growth in your namespace

[TIP]
====
*What to look for*:

* If the Gauge shows *green (0-5 nodes)*, your cluster has comfortable headroom
* *Yellow (5-10)* means you should start a procurement conversation this quarter
* *Red (10+)* means you need nodes sooner — initiate an emergency capacity request

In your own environment, replace the hard-coded 8-core assumption by setting `NODE_CPU` to your actual worker node allocatable CPU before running the dashboard script. This makes the thresholds meaningful for your specific infrastructure.
====

=== Part 6: Validate the Forecast

Let's cross-check the model against what's actually running. Run this command to count current pods:

Now run the full validation echo:

[source,bash,subs="+attributes",role=execute]
----
echo "Current pod count: $(oc get pods -n {namespace} --no-headers | wc -l)"
echo ""
echo "Forecast predicted (based on 30-day velocity):"
echo "This would require manual calculation from historical data"
----

[TIP]
====
*Facilitator Discussion Point*:

In production, you'd store daily snapshots of these metrics in a time-series database (Thanos/Prometheus long-term storage) to validate forecast accuracy over time.

*Accuracy improves with*:

* Longer historical data (90 days > 30 days)
* Seasonal adjustment (Black Friday, tax season, etc.)
* Deployment pattern clustering (some teams deploy weekly, others quarterly)

[IMPORTANT]
====
*The Forecasting Pitfall*:

Forecasting is only as good as your *resource requests*. If developers over-request by 10x, your forecast will tell you to buy 10x more hardware than you actually need.

This is why Module 3 (Developer Track - Right-Sizing) is critical. You must fix request accuracy BEFORE forecasting, or you'll forecast waste.
